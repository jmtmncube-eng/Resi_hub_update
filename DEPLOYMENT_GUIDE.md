```
            ┌──────────────────────────────────────┐
            │          Cloudflare (Free)             │
            │    CDN + DDoS Protection + SSL         │
            └───────────────┬──────────────────────┘
                            │
           ┌────────────────┼─────────────────┐
           │                                  │
  ┌────────▼────────┐              ┌──────────▼───────┐
  │  Vercel (Pro)   │              │ Railway (Hobby)   │
  │ React Frontend  │              │ Node.js Backend   │
  │ resihub.co.za   │              │ api.resihub.co.za │
  └─────────────────┘              └──────────┬───────┘
                                              │
                            ┌─────────────────┼──────────────┐
                            │                 │              │
                  ┌─────────▼──────┐  ┌───────▼──────┐  ┌──▼──────────┐
                  │  PostgreSQL    │  │   AWS S3     │  │  Sentry     │
                  │ (Railway DB)   │  │  af-south-1  │  │  (Errors)   │
                  └────────────────┘  └──────────────┘  └─────────────┘
```

### Why this stack
- **Vercel** — Zero-config React/Vite hosting, global CDN, automatic preview deployments
- **Railway** — Simple Node.js hosting with managed PostgreSQL, no infrastructure management
- **AWS S3 (af-south-1)** — Files stored in Cape Town for POPIA data residency
- **Cloudflare** — Free SSL, DDoS protection, CDN
- **Sentry** — Error tracking (free tier: 5,000 errors/month)

## 3. Security Requirements

- ✅ HTTPS enforced everywhere (Cloudflare handles SSL)
- ✅ JWT tokens — short-lived (24h access, 7d refresh)
- ✅ bcrypt password hashing (12 rounds in production)
- ✅ Zod validation on all API inputs
- ✅ Helmet.js security headers
- ✅ CORS locked to frontend origin
- ✅ Rate limiting on `/api/auth` endpoints
- ✅ File upload: MIME type + size validation
- ✅ Prisma parameterised queries (SQL injection prevention)

## 4. POPIA Compliance (South African Data Law)

The Protection of Personal Information Act (POPIA) applies to all personal data of South African residents.

### Key Requirements

| Requirement | Implementation |
|-------------|----------------|
| Data stored in SA | AWS S3 af-south-1 (Cape Town) |
| User consent | Registration consent checkbox |
| Right to access | `GET /api/profile` returns all personal data |
| Right to deletion | `DELETE /api/profile` — hard delete flow |
| Privacy policy | `/privacy-policy` page |
| Breach notification | Sentry alerts + manual reporting process |
| Minimum data collection | Only collect what's needed for residence management |

## 5. Environment Configuration

ResiHub runs as a Docker Compose stack on a single VPS (Postgres + backend +
nginx-served frontend), fronted by host nginx for TLS. All configuration lives
in a single **`.env`** file at the project root — gitignored, never committed.

### `.env` (project root)

`.env.example` is the committed template. On the VPS you don't hand-write this:
`deploy.sh` creates `.env` on first run, **generates strong JWT secrets** with
`openssl rand -hex 32`, and backfills the plain settings. Hand-edits always win
(deploy.sh only backfills keys that are absent).

```env
# ── Secrets — generated by deploy.sh on first run; never commit ──
JWT_SECRET=                       # openssl rand -hex 32
JWT_REFRESH_SECRET=               # openssl rand -hex 32
POSTGRES_PASSWORD=resihub         # change for production

# ── Tokens ──
JWT_EXPIRES_IN=24h
JWT_REFRESH_EXPIRES_IN=7d

# ── App ──
NODE_ENV=production
FRONTEND_URL=https://resihub.athera.co.za
FRONTEND_PORT=3001                # host port nginx proxies to
VITE_API_BASE_URL=/api            # baked into the frontend build
```

Vite env vars (`VITE_*`) are **inlined at build time** — `docker-compose.prod.yml`
passes them as build args into `Dockerfile.frontend.prod`, so they must be set
in `.env` *before* the image is built.

## 6. Deployment Steps

### One-line VPS deploy

The VPS only ever needs **one command**, run from the project root:

```bash
./deploy.sh                 # pull + build + migrate — keeps data
SEED=1 ./deploy.sh          # also reseed the DB (DESTRUCTIVE — wipes users)
```

`deploy.sh` is self-sufficient. It:
1. hard-resets to `origin/main` (discards stray edits on the VPS)
2. ensures `.env` exists — generates JWT secrets on first run, backfills settings
3. rebuilds the Docker images from scratch (`--no-cache`)
4. recreates the containers
5. waits for the backend to come up healthy (~3 min budget for the ts-node compile)
6. applies database migrations (baselines a pre-existing db-push-era DB first)
7. optionally reseeds (`SEED=1` only)
8. reloads host nginx

First time only: `chmod +x deploy.sh`

### Dev vs prod compose split

| File | Loaded when | Frontend it runs |
|------|-------------|------------------|
| `docker-compose.yml` | always (base) | — (shared config only) |
| `docker-compose.override.yml` | auto-loaded by plain `docker compose` | Vite dev server (HMR) on `:3000` |
| `docker-compose.prod.yml` | `deploy.sh` sets `COMPOSE_FILE` explicitly | static build served by nginx on `:3001` |

Local dev is just `docker compose up -d` (override auto-loads). `deploy.sh`
exports `COMPOSE_FILE="docker-compose.yml:docker-compose.prod.yml"`, which
disables the dev override and uses the prod frontend instead.

> The base compose deliberately keeps `command`/`ports`/`volumes` **out** of
> the `frontend` service — Compose *concatenates* those lists across files, so
> putting them only in the override files avoids a dev+prod port clash.

### Database migrations

Migrations are versioned (`prisma migrate deploy`), not `db push`. `deploy.sh`
runs them automatically in step 6. To run by hand inside the stack:

```bash
docker compose exec backend npx prisma migrate deploy
docker compose exec backend npx prisma migrate status   # verify
```

## 8. Database Backups

### Railway (Automatic)
- Daily automated snapshots (retained 7 days on Hobby plan)
- Manual snapshot before every major migration

### Manual Backup
```bash
pg_dump $DATABASE_URL > backup_$(date +%Y%m%d).sql
```
